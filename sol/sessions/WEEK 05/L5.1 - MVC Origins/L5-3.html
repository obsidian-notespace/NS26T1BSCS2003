1. CRUD Definition and Origins

Definition

CRUD is an acronym that stands for Create, Read, Update, Delete – the four fundamental operations that can be performed on persistent data. The term was originally defined in the context of databases, not the web. It reflects the lifecycle of data in any data model: data is first created, then read multiple times, occasionally updated, and finally deleted when no longer needed. In modern web applications, CRUD operations are mapped to HTTP methods and form the basis of most RESTful APIs.

Explanation

Motivation: Before CRUD was formalised, developers implemented ad‑hoc operations for each data entity, leading to inconsistent interfaces. By distilling common operations into four standard actions, CRUD provides a uniform vocabulary that simplifies design, communication, and implementation. Whether you are building a student gradebook, an e‑commerce site, or a social network, you will inevitably need to create, read, update, and delete records.

Historical context: The term CRUD emerged from database and software engineering communities long before the web. It was later adopted by web frameworks (e.g., Ruby on Rails, Django) to describe resource‑oriented controllers. The lecture emphasises that “originally defined in the context of databases, as such CRUD has nothing to do with the web.” This historical fact is important because it explains why CRUD is not inherently tied to HTTP – the mapping is a convenient adaptation.

First principles: Every piece of persistent data has a lifecycle. When you first store it (e.g., insert a row into a table), you perform a Create operation. Later, you may retrieve that data (Read) to display it to a user or use it in calculations. If the data changes (e.g., a student’s address changes), you Update it. Finally, when the data is no longer needed (e.g., a student graduates), you Delete it. These four operations are complete: any modification to a data store can be expressed as a combination of Create, Read, Update, and Delete.

Mechanism: In a relational database, CRUD maps directly to SQL statements: CREATE → INSERT, READ → SELECT, UPDATE → UPDATE, DELETE → DELETE. In web MVC, CRUD maps to Controller actions and HTTP methods: Create → POST, Read → GET, Update → PUT/PATCH, Delete → DELETE. The mapping is not one‑to‑one in the sense that Read may have two variants (list all resources vs show a single resource), and Update may be split into PUT (full replace) and PATCH (partial update).

Concrete example (from lecture transcript): The gradebook application requires creating new students, reading the list of students or a single student’s details, updating a student’s marks or address, and deleting a student who has graduated. Each of these corresponds to one of the CRUD operations.

Properties & characteristics: (1) Create is not idempotent – submitting the same creation request twice may create two records. (2) Read is idempotent and safe – repeated reads do not change state. (3) Update (PUT) is idempotent; PATCH may not be. (4) Delete is idempotent – deleting an already deleted record has no effect. (5) The operations can be composed – for example, “upsert” (update or insert) is a combination of Update and Create.

Relationship to other concepts: CRUD is the foundation of most data‑driven applications. In MVC, CRUD operations are implemented by Controller actions that interact with the Model. The View is used to present forms for Create/Update and to display data for Read.

Variations & special cases: (1) Soft delete – instead of physically deleting a record, a flag (e.g., `is_deleted`) is set to true; the record is excluded from reads but can be restored. (2) Batch operations – creating, updating, or deleting multiple records in one request. (3) Read operations may include filtering, sorting, pagination, and aggregation (e.g., summaries, charts).

Significance: CRUD is universal. Almost every application that stores data uses these four operations. Understanding CRUD helps you design consistent APIs, optimise database performance, and communicate requirements clearly.

Exam relevance: AQ5.3 Question 1 directly asks which operations are primitive database operations involved with MVC (all of Delete, Create, Update, Read). Questions about CRUD appear frequently in exams, often asking to identify which operation corresponds to a given scenario or to match CRUD to SQL/HTTP.

Intuition / Mental Model

To‑do list analogy: You create a new task (Create), you look at your list (Read), you mark a task as completed (Update), you remove a task (Delete). These four actions cover everything you can do with a to‑do list.

Library book analogy: A library acquires a new book (Create). Patrons read the book (Read). The library updates the book’s condition or location (Update). The library discards an old book (Delete).

Key Definitions

  • Create: The operation that inserts a new record into a data store. The record must not already exist (uniqueness constraints may apply).
  • Read: The operation that retrieves records from a data store, possibly with filtering, sorting, or aggregation.
  • Update: The operation that modifies an existing record. The record must exist before the update.
  • Delete: The operation that removes a record from a data store. After deletion, the record is no longer accessible via normal read operations.
  • Idempotent: An operation that can be applied multiple times without changing the result beyond the first application (e.g., DELETE is idempotent).

Formulas / Formal Notation

Not applicable – CRUD is a conceptual pattern. However, the mapping to SQL can be expressed as:

\[ \text{Create: } \text{INSERT INTO } R \text{ VALUES } (v_1, v_2, \dots) \]

\[ \text{Read: } \text{SELECT } \text{columns} \text{ FROM } R \text{ WHERE } \text{condition} \]

\[ \text{Update: } \text{UPDATE } R \text{ SET } \text{column} = \text{value} \text{ WHERE } \text{condition} \]

\[ \text{Delete: } \text{DELETE FROM } R \text{ WHERE } \text{condition} \]

Step-by-Step Procedure

Algorithm: Implementing a CRUD interface for a resource

  1. Step 1: Identify the resource (e.g., Student, Product, Post).
  2. Step 2: Define the data schema (attributes, data types, constraints).
  3. Step 3: Implement the Create operation: validate input, check for duplicates, insert into database, return the new record’s identifier.
  4. Step 4: Implement Read operations: list (with pagination, filtering) and detail (by identifier).
  5. Step 5: Implement Update operation: validate input, check that the record exists, modify fields, persist changes.
  6. Step 6: Implement Delete operation: check existence, remove record (or soft delete).
  7. Step 7: Expose these operations via an API (HTTP endpoints) or a user interface.

Common Mistakes & Misconceptions

  • Misconception: “CRUD was invented for web applications.”
    Why it's wrong: The lecture explicitly states that CRUD was originally defined in the context of databases and has nothing to do with the web. It was later co‑opted for web APIs.
    Correction: CRUD is a data lifecycle pattern that predates the web.
    How to avoid: Remember that CRUD operations are about data, not about HTTP.
  • Misconception: “Read operation always returns all records.”
    Why it's wrong: Read operations often include filtering, pagination, and selection of specific fields to avoid performance issues.
    Correction: Read can be parameterised to return subsets of data.
    How to avoid: Always consider adding query parameters for filtering and pagination in APIs.
  • Misconception: “Delete physically removes data from the database.”
    Why it's wrong: Many systems implement “soft delete” where a flag is set, and the record is not physically removed to preserve audit trails or allow restoration.
    Correction: Delete may be logical (soft) or physical; the term “delete” refers to making the record inaccessible via normal reads.
    How to avoid: Specify whether delete is soft or hard in your API documentation.

Edge Cases & Corner Cases

  • Edge case – Create with missing mandatory fields: The operation should fail with a validation error; the database may enforce NOT NULL constraints.
  • Edge case – Read on empty collection: Should return an empty list, not an error.
  • Edge case – Update on non‑existent record: Should return a 404 Not Found error (or 409 Conflict depending on design).
  • Edge case – Delete on non‑existent record: Idempotent, so returning 204 No Content or 404 Not Found are both acceptable; many APIs return 404 for idempotency clarity.

Real-World Application

GitHub Issues: Create an issue (POST /repos/.../issues), Read issues (GET /repos/.../issues), Update an issue (PATCH /repos/.../issues/1), Delete an issue (DELETE /repos/.../issues/1).

Amazon product catalogue: Create a new product (admin), Read product details (any user), Update price/stock (admin), Delete product (admin).

2. Create Operation

Definition

The Create operation inserts a new record into a data store. It requires that the new record does not already exist (to avoid conflicts) and that all mandatory fields are provided. The lecture highlights that while creating a new entry, one must check within the database to avoid duplicate records, and distinguish between mandatory and optional fields (e.g., name is mandatory, mobile number may be optional).

Explanation

Motivation: Without a standardised Create operation, applications would have inconsistent ways of adding data. The Create operation encapsulates the logic of validating input, enforcing uniqueness, and persisting the new record.

First principles: Creating a new data entity involves three steps: (1) validation – ensure that all mandatory fields are present and that the data meets business rules (e.g., email format, age range); (2) conflict detection – check that a record with the same unique identifier does not already exist; (3) insertion – write the new record to the database.

Constraints: The lecture mentions “Must not already exist. Check within database to avoid conflicts.” This refers to uniqueness constraints. For example, a student should not be created with the same roll number as an existing student. Also, mandatory fields (name, roll number) must be provided, while optional fields (address, phone) may be omitted.

Mechanism: In a web application, the Create operation is typically invoked via a POST request to a collection URL (e.g., `POST /students`). The Controller receives the data, validates it, and calls a Model method (e.g., `Student.create(params)`). The Model then performs the database insert and returns the newly created record (often including an auto‑generated ID). The Controller then redirects to the “show” page or returns a success response.

Concrete example: Adding a new student to the gradebook. The user fills a form with name, roll number, date of birth, and optional address. The server checks that the roll number is not already used; if it is, it returns an error. If valid, it inserts a new row into the `students` table and returns the student’s ID.

Properties & characteristics: (1) Not idempotent – submitting the same creation request twice will (if no duplicate prevention) create two records. (2) Often returns the identifier of the newly created resource. (3) May involve auto‑incrementing primary keys or UUIDs.

Relationship to other concepts: In RESTful APIs, the Create operation is implemented with POST to the collection endpoint. In MVC, the Controller action is typically named `create` (or `store` in some frameworks).

Variations & special cases: (1) Bulk create – inserting multiple records in one request. (2) Upsert – create if not exists, else update. (3) Create with nested resources – e.g., creating a comment for a specific post (`POST /posts/1/comments`).

Intuition / Mental Model

Creating a new record is like filling out a registration form: you provide required information (mandatory fields), and the system checks that you are not already registered (duplicate prevention). If everything is fine, you are added to the system.

Key Definitions

  • Mandatory field: A field that must have a value for the record to be created; the database typically enforces NOT NULL.
  • Optional field: A field that can be left empty; the database allows NULL.
  • Unique constraint: A rule that no two records can have the same value for a given field (or combination of fields).
  • Auto‑increment: A mechanism that automatically generates a unique integer for a primary key when a new record is inserted.

Step-by-Step Procedure

  1. Step 1: Receive input data (from form or API).
  2. Step 2: Validate all mandatory fields are present and meet format requirements.
  3. Step 3: Check for uniqueness of any unique fields (e.g., email, username).
  4. Step 4: If validation fails, return error response and re‑render the creation form with error messages.
  5. Step 5: If validation passes, create a new Model instance and persist to the database.
  6. Step 6: Return a success response (e.g., redirect to the new record’s detail page).

Common Mistakes & Misconceptions

  • Misconception: “Create operation always uses POST.” – While POST is standard, PUT can also be used for create if the client knows the resource ID (idempotent creation).
  • Misconception: “Duplicate checks are optional.” – Without them, you risk data integrity issues; they are essential for unique identifiers.

Real-World Application

Signing up for a new account on a website: you provide email and password (mandatory), optional profile picture, and the system checks if the email is already taken.

  • Typical exam trap: Thinking that Create is idempotent – it is not.

* Create inserts a new record; it requires validation and duplicate checks.
* Mandatory fields must be provided; optional fields may be omitted.
* Create is typically implemented with POST in web APIs.
* The operation returns the identifier of the newly created resource.

3. Read Operation

Definition

The Read operation retrieves data from a data store. It can be simple (fetch a single record by identifier) or complex (list all records with filtering, sorting, pagination, and aggregation). The lecture gives examples: “Get a list of students”, “Summarise number of students, age distribution, geographic locations”, “Plot histograms of marks”. These are all Read operations, even though they involve computation (aggregation) – they do not modify data.

Explanation

Motivation: Reading is the most frequent operation in many applications (e.g., news websites, social media feeds). Optimising read operations is critical for performance.

First principles: A Read operation does not change the state of the database. It may perform complex queries, joins, aggregations, and transformations, but the underlying data remains unchanged. The results are presented to the user (via a View) or returned as data (JSON).

Mechanism: In web MVC, Read operations are typically invoked via GET requests. The Controller action (e.g., `index` for list, `show` for single) calls Model methods that execute SELECT queries. The results are passed to a View for rendering.

Concrete example (from lecture): In the gradebook, a teacher may want to see a histogram of marks for a particular course. This requires reading all enrolment records for that course, extracting the marks, and computing the distribution. The operation is a Read (no data is changed), but it involves aggregation.

Properties & characteristics: (1) Safe – no side effects. (2) Idempotent – repeated reads return the same result (unless data changes between reads). (3) Cacheable – responses can be cached by browsers or CDNs. (4) Can be parameterised (filtering, sorting, pagination).

Intuition / Mental Model

Reading is like looking at a report – you can view it many times without altering the source data. Summaries and charts are just different ways of presenting the same underlying data.

Key Definitions

  • Aggregation: A computation that combines multiple values (e.g., SUM, AVG, COUNT).
  • Histogram: A graphical representation of the distribution of numerical data.
  • Pagination: Dividing a large result set into smaller pages to improve performance and usability.

Step-by-Step Procedure

  1. Step 1: Receive query parameters (e.g., `?page=2&sort=name`).
  2. Step 2: Construct a query to the Model (e.g., `Student.query.filter(...).order_by(...).paginate(...)`).
  3. Step 3: Execute the query and retrieve results.
  4. Step 4: If aggregation is needed, perform the aggregation in the database (using GROUP BY, COUNT, etc.) to avoid loading all data into memory.
  5. Step 5: Pass the results to the View and render the response.

Common Mistakes & Misconceptions

  • Misconception: “Read operations cannot perform calculations.” – They can (aggregation, formatting) as long as they don’t modify data.
  • Misconception: “All reads should return all fields.” – Often, only a subset of fields is needed (projection).

Real-World Application

Google Analytics dashboard: reads data from tracking logs, computes summaries and charts, and displays them without altering the raw data.

Activity Questions (from AQ5.3)

No AQ5.3 questions isolate Read, but Question 1 includes Read as a primitive operation.

Exam Tips & Points to Remember

  • Most frequently tested aspect: Read operations are safe and idempotent.
  • Typical exam trap: Thinking that generating a histogram is an Update – it is a Read.

Key Takeaways

* Read retrieves data without modifying it.
* Read operations can include filtering, sorting, pagination, and aggregation.
* Summaries and charts are examples of aggregated reads.
* Read is the most common operation in many applications.
* Use GET for read operations in web APIs.

4. Update Operation

Definition

The Update operation modifies an existing record in a data store. The lecture gives examples: “Change of address”, “Update marks”, “Change start date of course”. Update requires that the record already exists; otherwise, the operation should fail (or, in some designs, create the record – upsert).

Explanation

Motivation: Data changes over time; Update allows correction and evolution of records without recreating them from scratch.

First principles: Updating a record involves: (1) identifying the record (usually by primary key), (2) validating the new values, (3) applying the changes to the database. Updates can be full (replace all fields) or partial (only specified fields).

Mechanism: In web MVC, Update is typically implemented with PUT (full replace) or PATCH (partial update). The Controller receives the updated data, finds the record, validates, and calls Model methods to persist changes.

Concrete example: A student changes their address. The application sends a PUT request to `/students/5` with the new address. The server updates the record and returns the updated student data.

Properties & characteristics: (1) PUT is idempotent; PATCH is not necessarily. (2) Update operations should be preceded by authorisation checks (e.g., only the student or an admin can update).

Intuition / Mental Model

Updating is like editing a document – you change some words, save the file, and the old version is replaced.

Key Definitions

  • Full update (PUT): Replaces the entire resource with the provided representation; omitted fields become null (or default).
  • Partial update (PATCH): Modifies only the fields specified in the request; other fields remain unchanged.

Step-by-Step Procedure

  1. Step 1: Receive identifier and new data.
  2. Step 2: Retrieve the existing record from the database.
  3. Step 3: If the record does not exist, return 404 Not Found.
  4. Step 4: Validate the new data (e.g., marks between 0 and 100).
  5. Step 5: Apply the changes (for PUT, replace all fields; for PATCH, merge).
  6. Step 6: Persist the changes to the database.
  7. Step 7: Return the updated record (or a success status).

Common Mistakes & Misconceptions

  • Misconception: “PUT and PATCH are interchangeable.” – They are not; PUT requires the client to send the full resource, while PATCH sends only differences.
  • Misconception: “Update should always return the full updated resource.” – It may return just a success status; but returning the resource is common.

Real-World Application

Editing a Wikipedia article: you change some text, and the system updates the page while preserving the revision history.

Activity Questions (from AQ5.3)

No AQ5.3 questions isolate Update, but Question 1 includes Update as a primitive operation.

Exam Tips & Points to Remember

  • Most frequently tested aspect: Difference between PUT and PATCH.
  • Typical exam trap: Using PUT for partial updates – may unintentionally clear omitted fields.

Key Takeaways

* Update modifies an existing record.
* PUT replaces the entire resource (idempotent); PATCH applies partial changes.
* Always validate update data and check authorisation.
* If the record does not exist, return 404 (or use upsert).

5. Delete Operation and Archiving

Definition

The Delete operation removes a record from a data store. The lecture discusses that deletion may be necessary for graduated students or to keep the database size manageable. However, it also mentions archiving as an alternative: moving records to a different table or database instead of permanently deleting them. Archiving preserves data for historical analysis while keeping the live system lean.

Explanation

Motivation: Over time, databases accumulate large volumes of old data, which can degrade query performance. Deleting or archiving obsolete data is essential for maintaining system performance and reducing storage costs.

First principles: Physical delete removes the record from the database entirely; it cannot be undone (except from backups). Soft delete sets a flag (e.g., `deleted_at`) that causes the record to be excluded from normal reads; the data remains in the table and can be restored. Archiving moves the record to a separate table or database, effectively deleting it from the live system but retaining it for historical queries.

Mechanism: In web MVC, Delete is typically invoked via a DELETE request to a resource URL (e.g., `DELETE /students/5`). The Controller finds the record, performs authorisation checks, and then calls the Model’s delete method. For soft delete, the Model updates a flag; for hard delete, it issues a DELETE SQL statement.

Concrete example (from lecture): “Maybe a graduated student, I remove them from the database. Usually, you would prefer to do something like archiving.” The lecture explains that when the number of students grows to lakhs or crores, searching becomes slow. Archiving moves old students to a different table, keeping the live table smaller and faster.

Properties & characteristics: (1) DELETE is idempotent – deleting an already deleted record has no effect. (2) Hard delete is irreversible; soft delete allows restoration. (3) Archiving is a form of logical deletion that preserves data for analytics.

Intuition / Mental Model

Deleting is like throwing away a document. Archiving is like moving it to a storage room – you cannot see it in the office, but you can retrieve it if needed.

Key Definitions

  • Hard delete: Physically removes the record from the database; the space may be reclaimed.
  • Soft delete: Marks the record as deleted (e.g., setting a `deleted_at` timestamp) without removing it; normal queries exclude deleted records.
  • Archive: Moving data to a separate storage location (another table, another database, or offline storage) to reduce load on the live system.

Step-by-Step Procedure

  1. Step 1: Receive identifier of record to delete.
  2. Step 2: Retrieve the record (or check existence).
  3. Step 3: Authorise the deletion (e.g., only admin or owner).
  4. Step 4: If using soft delete, set a flag and save; if hard delete, execute DELETE statement.
  5. Step 5: If archiving, copy the record to an archive table before deletion.
  6. Step 6: Return a success response (e.g., 204 No Content).

Common Mistakes & Misconceptions

  • Misconception: “Delete is irreversible, so never use it.” – Soft delete or archiving can provide safety.
  • Misconception: “Archiving is the same as backing up.” – Archiving is for active data management; backups are for disaster recovery.

Real-World Application

Email providers: “Trash” folder is a soft delete; after 30 days, emails are hard deleted. Archiving moves emails to an “All Mail” folder (still accessible but not in Inbox).

Activity Questions (from AQ5.3)

No AQ5.3 questions isolate Delete, but Question 1 includes Delete as a primitive operation.

Exam Tips & Points to Remember

  • Most frequently tested aspect: Distinction between hard delete, soft delete, and archiving.
  • Typical exam trap: Thinking that DELETE is not idempotent – it is.

Key Takeaways

* Delete removes data; it can be hard (physical) or soft (logical).
* Archiving moves data to a separate storage location for historical retention.
* DELETE is idempotent – calling it twice has the same effect as once.
* Always authorise delete operations to prevent accidental or malicious data loss.

6. Read‑heavy vs Write‑heavy Databases

Definition

Databases and storage systems can be optimised for different workload patterns. A read‑heavy system is one where read operations (SELECT) vastly outnumber write operations (INSERT, UPDATE, DELETE). A write‑heavy system is one where writes dominate. The lecture gives examples: a course catalogue with many reads but few writes is read‑heavy; a security archive log that is continuously written but rarely read is write‑heavy.

Explanation

Motivation: Optimising a database for reads requires different strategies (indexes, caching, replication) than optimising for writes (which may favour append‑only storage, write‑ahead logs, or sharding). Choosing the wrong optimisation can lead to poor performance.

First principles: In a read‑heavy system, the goal is to minimise query latency. Techniques include: adding indexes on frequently queried columns, using materialised views, caching query results in memory (e.g., Redis), and using read replicas to distribute load. In a write‑heavy system, the goal is to maximise write throughput. Techniques include: using append‑only data structures (e.g., log‑structured merge trees), batching writes, avoiding indexes that slow down inserts, and using sharding to distribute writes across many nodes.

Concrete examples (from lecture): “Read‑heavy: lots of reading, very little writing or creating” – e.g., a course catalogue for the IITM BSc degree where students browse courses. “Write‑heavy: security archive logs” – e.g., system logs that must be recorded for compliance but are rarely queried.

Properties & characteristics: (1) Read‑heavy systems benefit from caching and indexes. (2) Write‑heavy systems benefit from append‑only storage and minimal indexing. (3) Some systems are balanced; they require careful tuning.

Relationship to other concepts: Understanding workload patterns helps in choosing the right database technology (SQL vs NoSQL, column‑oriented vs row‑oriented).

Intuition / Mental Model

A library is read‑heavy: many people browse books (reads), but new books are added infrequently (writes). A transaction log at a bank is write‑heavy: every transaction is recorded (write), but the log is rarely read except for audits.

Key Definitions

  • Read‑heavy workload: The ratio of read operations to write operations is high (e.g., 1000:1).
  • Write‑heavy workload: The ratio of write operations to read operations is high (e.g., 100:1).
  • Index: A data structure that improves the speed of read operations at the cost of slower writes.
  • Cache: A temporary storage layer that stores frequently read data to reduce database load.
  • Sharding: Partitioning a database across multiple servers to distribute write load.

Step-by-Step Procedure

Algorithm: Optimising for read‑heavy workload

  1. Step 1: Identify the most frequent queries (e.g., “find student by roll number”).
  2. Step 2: Add indexes on columns used in WHERE clauses.
  3. Step 3: Implement caching (e.g., store query results in Redis for a few minutes).
  4. Step 4: Use read replicas to offload read queries from the primary database.
  5. Step 5: Denormalise data where appropriate to reduce joins.

Diagrams / Tables

Characteristic Read‑heavy Write‑heavy
Example Catalogue, news site Logging, sensor data, social media posts
Optimisation focus Low query latency High write throughput
Indexing Many indexes Few indexes (or specialised indexes)
Caching Very effective Less effective (cache invalidates quickly)
Database choice MySQL, PostgreSQL, read replicas Cassandra, InfluxDB, log‑structured engines

Common Mistakes & Misconceptions

  • Misconception: “All databases should be optimised for reads.” – Write‑heavy systems need different optimisations; over‑indexing can kill write performance.
  • Misconception: “Write‑heavy means you never read.” – Even write‑heavy systems have occasional reads; the ratio matters.

Real-World Application

Twitter feed: Reading tweets is read‑heavy; but the act of posting a tweet is a write. Twitter uses a combination of caching and sharding to handle both.

Stock ticker: Extremely write‑heavy (thousands of price updates per second), but also read‑heavy (many clients watching). Specialised databases like kdb+ are used.

  • (C) Machine Learning models – training is write‑heavy (writing model parameters), but the statement is ambiguous; however, compared to security logs, training is not continuous and often involves both reads and writes. Security logs are the clear example of write‑heavy.
  • (D) Reading emails – clearly read‑heavy.

Concept tested: Understanding the definition of write‑heavy workloads and identifying real‑world examples.

Exam Tips & Points to Remember

  • Most frequently tested aspect: Recognising read‑heavy vs write‑heavy scenarios.
  • Typical exam trap: Assuming that “write‑heavy” means no reads – even write‑heavy systems have occasional reads.

Key Takeaways

* Read‑heavy workloads require indexes, caching, and read replicas.
* Write‑heavy workloads require minimising indexes, using append‑only storage, and sharding.
* Security archive logs are a canonical write‑heavy example.
* The workload pattern influences database technology choices.

7. APIs and CRUD as First‑Level API

Definition

An API (Application Programming Interface) defines a standard way for clients to communicate with a server. The lecture explains that CRUD provides a good set of functions to build a basic API because the operations are well‑defined and map naturally to HTTP methods. A CRUD‑based API allows clients to create, read, update, and delete resources without needing to know the server’s implementation details.

Explanation

Motivation: Without a standardised API, every client would need to understand the server’s internal database schema and logic. An API abstracts away the implementation, allowing the server to change (e.g., switch from MySQL to PostgreSQL) without breaking clients.

First principles: An API defines the “contract” between client and server: what requests are allowed, what parameters they accept, and what responses they return. A CRUD API is a simple contract that aligns with the natural lifecycle of data. The client sends an HTTP request with a method (GET, POST, PUT, DELETE) to a specific URL, and the server performs the corresponding CRUD operation.

Mechanism: The lecture describes: “If you want my server to create an object, you send a request like this; if you want to read an object, you will send a request like this.” This is exactly how RESTful APIs work. The server defines endpoints (e.g., `/students`) and methods, and clients adhere to that specification.

Concrete example: A simple API for a student database might have: `GET /students` (list), `GET /students/5` (retrieve student 5), `POST /students` (create new student), `PUT /students/5` (update student 5), `DELETE /students/5` (delete student 5). The client does not need to know whether the server uses SQLite or PostgreSQL; it only needs to know the API.

Properties & characteristics: (1) Abstraction – clients are decoupled from implementation. (2) Standardisation – CRUD provides a familiar pattern. (3) CRUD is typically considered the “first level API” – a starting point that can be extended with additional custom actions (e.g., `activate`, `archive`).

Relationship to other concepts: In MVC, the API endpoints are implemented by Controller actions. The Model provides the data persistence, and the View (if any) is used to format responses (e.g., JSON).

Variations & special cases: (1) GraphQL – an alternative to CRUD APIs that allows clients to request specific fields. (2) RPC‑style APIs – use POST for everything, which is less semantic.

Intuition / Mental Model

The API is like a vending machine – you press a button (HTTP method and URL), and the machine gives you a product (response). You don’t need to know how the machine is stocked or wired; you just follow the instructions on the front.

Key Definitions

  • API endpoint: A specific URL path that exposes a particular operation.
  • Resource: A conceptual entity that the API manages (e.g., Student, Product).
  • Contract: The agreement between client and server specifying request and response formats.

Step-by-Step Procedure

  1. Step 1: Identify the resources your application manages.
  2. Step 2: For each resource, define the CRUD operations needed.
  3. Step 3: Map each operation to an HTTP method and URL pattern (e.g., `POST /resources`).
  4. Step 4: Specify the request format (e.g., JSON body, form data) and response format.
  5. Step 5: Implement the Controller actions that perform the operations.
  6. Step 6: Document the API for clients.

Common Mistakes & Misconceptions

  • Misconception: “APIs must be RESTful and use CRUD.” – Not necessarily; GraphQL and RPC are alternatives.
  • Misconception: “CRUD covers all possible operations.” – It does not cover custom actions like “approve”, “reject”, “send email”. Those can be added as additional endpoints (e.g., `POST /orders/1/approve`).

Real-World Application

GitHub API: Provides CRUD operations for repositories, issues, comments, etc. Also provides additional actions like “star” a repository (`PUT /user/starred/:owner/:repo`).

Activity Questions (from AQ5.3)

No AQ5.3 questions directly test APIs, but Question 1 establishes CRUD as primitive operations.

Exam Tips & Points to Remember

  • Most frequently tested aspect: CRUD as a foundation for building APIs.
  • Typical exam trap: Thinking that CRUD is the only way to design an API.

Key Takeaways

* An API abstracts the implementation, allowing clients to communicate via standard requests.
* CRUD provides a natural set of operations for a first‑level API.
* Each CRUD operation maps to an HTTP method and URL pattern.
* Custom actions (beyond CRUD) can be added as additional endpoints.
* The client only needs to know the API, not the server’s internals.
