1. Actions vs Controllers

Definition

An action is a single operation that the system performs in response to a user request (e.g., “create a student”, “list all courses”). A controller is a logical grouping of related actions. The lecture states: “Can actions be grouped together logically? Controller!” In other words, a controller is a class (or module) that contains several action methods that all operate on the same resource or share a common purpose.

Explanation

Motivation: Without controllers, actions would be scattered across many unrelated functions or classes, making the application hard to navigate and maintain. Grouping related actions (e.g., all actions that manipulate “photos”) into a single controller improves organisation, reduces duplication, and makes the code more predictable.

First principles: Every action is a function that handles a specific request. When you have multiple actions that operate on the same type of data (e.g., a `Photo` resource), it makes sense to put them together. For example, actions that create, read, update, and delete photos are all related to the “photo” concept. A controller named `PhotosController` can contain all these actions.

Mechanism: In web MVC frameworks, a controller is a class. Each public method is an action. The router maps URLs and HTTP methods to specific controller actions. For example, `GET /photos` might map to `PhotosController.index`, `GET /photos/1` to `PhotosController.show`, `POST /photos` to `PhotosController.create`, etc.

Concrete example (from lecture): The lecture mentions CRUD as a set of actions. Other actions could include “send email”, “update logs”, “send alert on WhatsApp/Telegram”. If you have many actions related to user notifications, you could group them into a `NotificationsController`. This is a logical grouping.

Properties & characteristics: (1) A controller typically corresponds to a resource (e.g., Student, Product, Order). (2) Actions within a controller often share common setup code (e.g., authentication, authorisation) via before filters. (3) Controllers promote reusability – common logic can be extracted into private methods.

Relationship to other concepts: This lecture builds on the action concept introduced in L5.1 and L5.2. The controller is the “container” for actions, and the router maps requests to controller actions.

Variations & special cases: Some frameworks (like Flask) do not have a built‑in controller class; you can group routes by using Blueprints or by organising functions into modules. This is still a form of logical grouping.

Significance: Grouping actions by controller is a standard practice in MVC frameworks. It makes the codebase more navigable and maintainable, especially as the application grows.

Exam relevance: AQ5.4 does not directly test this distinction, but it is foundational for understanding resource controllers.

Intuition / Mental Model

A controller is like a department in a company. The “Sales” department handles all actions related to sales (create order, update order, cancel order). The “HR” department handles employee‑related actions. Each department (controller) groups related tasks (actions).

Key Definitions

  • Action: A method in a controller that handles a specific type of request.
  • Controller: A class that groups related actions, usually corresponding to a resource.
  • Resource: A conceptual entity (e.g., Photo, User) that the controller manages.
  • Logical grouping: Organising code based on common purpose or domain.

Formulas / Formal Notation

Not applicable.

Step-by-Step Procedure

Algorithm: Deciding how to group actions into controllers

  1. Step 1: Identify the main resources in your application (e.g., Student, Course, Enrolment).
  2. Step 2: For each resource, list the actions needed (typically CRUD actions).
  3. Step 3: Create a controller for each resource (e.g., `StudentsController`).
  4. Step 4: Place all actions that operate on that resource inside the controller.
  5. Step 5: For actions that do not belong to a single resource (e.g., dashboard, search), create utility controllers (e.g., `HomeController`, `SearchController`).

Verification: The application should be organised so that a developer can quickly locate the action that handles a given URL by looking at the controller name.

Common Mistakes & Misconceptions

  • Misconception: “A controller can only contain CRUD actions.” – Controllers can contain any actions, including custom ones (e.g., `approve`, `archive`).
  • Misconception: “There must be exactly one controller per resource.” – You can have multiple controllers that interact with the same resource (e.g., an admin controller and a public controller).

Edge Cases & Corner Cases

  • Edge case – Actions that belong to multiple resources: For example, “enrol student in course” involves both Student and Course. It is often placed in an `EnrolmentsController` or `RegistrationsController`.

2. Resource Controller (Laravel Example)

Definition

A resource controller is a pre‑defined controller pattern that provides default actions for managing a resource. The lecture uses Laravel (a PHP framework) as an example. Laravel’s resource controller automatically includes seven actions: index, create, store, show, edit, update, destroy. By associating a resource controller with a model (e.g., “Photo”), the developer gets a standard set of actions without having to define each one manually.

Explanation

Motivation: Most resources require the same set of operations: listing, creating, showing, editing, updating, and deleting. A resource controller eliminates boilerplate code by providing these actions out‑of‑the‑box. Developers can then override or customise only the actions that need special behaviour.

First principles: The seven actions are:

  • index – display a list of resources (GET request).
  • create – show the form to create a new resource (GET).
  • store – save a new resource (POST).
  • show – display a single resource (GET).
  • edit – show the form to edit an existing resource (GET).
  • update – update an existing resource (PUT/PATCH).
  • destroy – delete a resource (DELETE).

Mechanism: In Laravel, you define a resource route: `Route::resource('photos', 'PhotoController')`. This single line creates routes for all seven actions, mapping URLs and HTTP methods to the controller actions. The controller class can be generated with `php artisan make:controller PhotoController --resource`, which creates stub methods for each action.

Concrete example (from lecture): The lecture shows a screenshot from Laravel documentation listing the seven actions along with their HTTP verbs and URLs. For a “photo” resource: `GET /photos` → index; `GET /photos/create` → create; `POST /photos` → store; `GET /photos/{photo}` → show; `GET /photos/{photo}/edit` → edit; `PUT/PATCH /photos/{photo}` → update; `DELETE /photos/{photo}` → destroy.

Properties & characteristics: (1) Convention over configuration – you follow naming conventions and get automatic routing. (2) The `create` and `edit` actions are for displaying forms; they are separate from `store` and `update` which actually persist data. (3) The resource controller can be customised (e.g., adding extra actions like `restore`).

Relationship to other concepts: This is a concrete implementation of the “group actions by controller” idea. The resource controller standardises the CRUD operations into seven named actions.

Variations & special cases: (1) Some frameworks (e.g., Ruby on Rails) have a similar concept with `resources :photos` in routes. (2) In Flask, you can simulate resource controllers by creating a class and using method views.

Significance: Resource controllers are a major productivity booster. They enforce consistency across the application and reduce the chance of missing essential actions.

Exam relevance: AQ5.4 does not directly ask about Laravel, but the concept of mapping HTTP methods to actions is tested in other questions (e.g., Q3 on storing an image).

Intuition / Mental Model

The resource controller is like a pre‑built toolkit for a specific job. If you are building a house, you don’t design every hammer from scratch – you use a standard toolkit. Similarly, for a data resource, you use a standard resource controller with pre‑defined actions.

Key Definitions

  • Resource route: A single routing declaration that generates all standard CRUD routes for a resource.
  • Convention over configuration: A design philosophy where the framework assumes sensible defaults, reducing the need for explicit configuration.
  • Stub method: An empty method body generated by the framework that you fill with your implementation.

Step-by-Step Procedure

Algorithm: Using a resource controller in Laravel

  1. Step 1: Run `php artisan make:controller PhotoController --resource`.
  2. Step 2: In `routes/web.php`, add `Route::resource('photos', 'PhotoController')`.
  3. Step 3: Implement each action method in `PhotoController` with your logic.
  4. Step 4: Create corresponding views (e.g., `index.blade.php`, `create.blade.php`, `show.blade.php`, `edit.blade.php`).
  5. Step 5: Test each endpoint by visiting the URLs.

  • Misconception: “`create` and `store` are the same action.” – `create` displays the form; `store` processes the form submission.
  • Misconception: “Resource controllers can only be used for database tables.” – They can be used for any resource, even virtual ones.

Real-World Application

Laravel applications almost universally use resource controllers for their models (e.g., `PostController`, `UserController`). This standardisation makes it easy for new developers to understand the codebase.

Mapping HTTP Verbs to Controller Actions

Definition

The mapping between HTTP methods (verbs) and controller actions is a central part of RESTful web applications. The lecture explains that the HTTP verb (GET, POST, PUT, PATCH, DELETE) is used to convey the meaning of the request, while the URL provides the context (the resource). For a resource controller, each verb maps to a specific action: GET → index/show/create/edit; POST → store; PUT/PATCH → update; DELETE → destroy.

Explanation

Motivation: Using HTTP verbs semantically allows browsers, proxies, and caches to understand the intent of the request. For example, a GET request can be cached; a DELETE request should not be repeated automatically.

First principles: The HTTP specification defines methods with specific semantics. A RESTful API respects these semantics: GET for retrieval, POST for creation, PUT for full replacement, PATCH for partial update, DELETE for removal. The controller actions are designed to align with these semantics.

Mechanism: The router inspects both the HTTP method and the URL path. For the same URL path, different methods can map to different actions. For example, `GET /photos` maps to `index`, while `POST /photos` maps to `store`. `GET /photos/1` maps to `show`, while `PUT /photos/1` maps to `update`, and `DELETE /photos/1` maps to `destroy`.

Concrete example (from lecture): The lecture points out that the Laravel resource controller uses verbs as shown: GET for index, create, show, edit; POST for store; PUT/PATCH for update; DELETE for destroy. This is the standard RESTful mapping.

Properties & characteristics: (1) Idempotent methods (GET, PUT, DELETE) can be safely retried. (2) Safe methods (GET) should not change state. (3) POST is neither safe nor idempotent.

Relationship to other concepts: This mapping is the bridge between HTTP and the MVC controller layer. It is essential for designing RESTful APIs.

Intuition / Mental Model

The HTTP verb is like the command verb in a sentence: “GET me the photo” vs “POST this photo to the server”. The URL is the noun (“photo”). The combination tells the server exactly what to do.

Key Definitions

  • HTTP verb: The method part of an HTTP request (GET, POST, PUT, PATCH, DELETE, etc.).
  • Idempotent: An operation that can be applied multiple times without changing the result beyond the first application.
  • Safe method: An HTTP method that does not change the server state (GET, HEAD).

Step-by-Step Procedure

When a request arrives:

  1. Step 1: Extract the HTTP method (e.g., GET).
  2. Step 2: Extract the URL path (e.g., `/photos/5`).
  3. Step 3: Match the method and path against routing rules.
  4. Step 4: Determine the controller and action from the match.
  5. Step 5: Dispatch to the action.

Diagrams / Tables

See the table in Concept Card 2.

Common Mistakes & Misconceptions

  • Misconception: “Any HTTP method can be used for any action.” – Technically yes, but it violates RESTful semantics and may cause issues with caching and idempotency.
  • Misconception: “PUT and PATCH are interchangeable.” – PUT replaces the entire resource; PATCH applies partial changes.

Real-World Application

GitHub API uses this mapping: `GET /repos/:owner/:repo` (show), `POST /repos/:owner/:repo/issues` (create issue), `PATCH /repos/:owner/:repo` (update repo), `DELETE /repos/:owner/:repo` (delete repo).

4. Custom Actions Beyond CRUD

Definition

Not all actions fit neatly into CRUD. The lecture gives examples: “send email”, “update logs”, “send alert on WhatsApp/Telegram”. These are custom actions that do not correspond to creating, reading, updating, or deleting a specific resource. They can still be grouped into controllers, but they require additional routes and action methods.

Explanation

Motivation: Real‑world applications have side effects and auxiliary operations that are not resource‑oriented. For example, a “send reset password email” action does not correspond to a CRUD operation on the User resource; it is a custom action.

First principles: Custom actions are typically implemented as additional methods in a controller, and they are routed using non‑RESTful URL patterns. For example, you might have `POST /users/1/send_welcome_email`.

Mechanism: In most frameworks, you can add custom routes alongside the resource routes. For example, in Laravel: `Route::post('/users/{user}/send_email', [UserController::class, 'sendEmail'])`. In Flask, you simply add a new route decorator.

Concrete example (from lecture): Sending an alert on WhatsApp or Telegram – this might be an action that is triggered by a button in the view. The controller would call an external API to send the message. This action does not modify the database (or only logs that an alert was sent).

Properties & characteristics: (1) Custom actions are often non‑idempotent and use POST (to avoid accidental triggering via GET). (2) They may be grouped with the related resource controller (e.g., `UserController` contains `sendWelcomeEmail`). (3) They can also be placed in separate controllers (e.g., `NotificationsController`).

Relationship to other concepts: This extends the idea of grouping actions beyond the standard CRUD set.

Variations & special cases: Some frameworks support “resourceful” custom actions using special routing syntax (e.g., `resources :users do member { post :send_email } end` in Rails).

Significance: Recognising that not every action is CRUD helps developers design intuitive APIs and avoid forcing operations into unnatural CRUD shapes.

Exam relevance: AQ5.4 does not directly ask about custom actions, but it is an important concept for real‑world development.

Intuition / Mental Model

Think of a vending machine that not only dispenses products (CRUD) but also has a “report malfunction” button. That button is a custom action – it doesn’t create or delete a product; it sends a notification to the maintenance team.

Key Definitions

  • Custom action: An action that does not correspond to standard CRUD operations on a resource.
  • Member route: A route that acts on a specific resource instance (e.g., `/users/1/send_email`).
  • Collection route: A route that acts on the collection of resources (e.g., `/users/export`).

Step-by-Step Procedure

  1. Step 1: Identify a user need that is not CRUD (e.g., “send invoice”).
  2. Step 2: Choose a meaningful action name (e.g., `send_invoice`).
  3. Step 3: Add the action method to the appropriate controller.
  4. Step 4: Define a route for the action (usually POST to avoid accidental triggers).
  5. Step 5: Implement the logic (call external APIs, send emails, etc.).
  6. Step 6: Return an appropriate response (redirect, JSON, etc.).

Common Mistakes & Misconceptions

  • Misconception: “Custom actions should use GET.” – Unless they are idempotent and safe, use POST to prevent accidental execution (e.g., a web crawler following a GET link could trigger an email send).
  • Misconception: “Custom actions must be in separate controllers.” – They can be placed in the same controller if they relate to the same resource.

Real-World Application

Amazon’s “Track Package” action – not a CRUD operation on an order, but a custom action that queries a shipping carrier’s API.

Activity Questions (from AQ5.4)

No AQ5.4 questions specifically address custom actions.

Exam Tips & Points to Remember

  • Most frequently tested aspect: Recognising which HTTP method to use for custom actions (usually POST).
  • Typical exam trap: Using GET for actions that change state.

Key Takeaways

* Not all actions fit CRUD; custom actions are needed for side effects and auxiliary operations.
* Custom actions should typically use POST to avoid accidental triggers.
* They can be grouped with related resource controllers or placed in separate controllers.
* Routing for custom actions often uses nested URLs (e.g., `/users/{id}/send_email`).

5. URL, URI, and URN

Definition

A URI (Uniform Resource Identifier) is a string that identifies a resource. A URL (Uniform Resource Locator) is a type of URI that identifies a resource by its location (e.g., `https://example.com/photo.jpg`). A URN (Uniform Resource Name) is a type of URI that identifies a resource by name in a given namespace, independent of location (e.g., `urn:isbn:0451450523`). The lecture touches on these concepts in the context of web addresses.

Explanation

Motivation: Understanding the hierarchy (URI superset, URL and URN subsets) is important for correctly interpreting web addresses and designing APIs.

First principles: All URLs are URIs, but not all URIs are URLs. A URN is also a URI but does not provide a location. For example, an ISBN number identifies a book (URN) but does not tell you where to buy it; a URL tells you where to find it.

Mechanism: In web development, we almost always work with URLs. However, the term URI is sometimes used in specifications (e.g., `request.uri` in some frameworks).

Concrete example: `https://www.example.com/index.html` is a URL (and also a URI). `urn:isbn:0-486-27557-4` is a URN (and a URI).

Properties & characteristics: (1) URL includes protocol and location. (2) URN is persistent and location‑independent. (3) URI is the general term.

Intuition / Mental Model

Think of a person: a URL is like your home address (where you live); a URN is like your Social Security number (who you are). Both identify you, but in different ways.

Key Definitions

  • URI: Uniform Resource Identifier – a compact string of characters that identifies an abstract or physical resource.
  • URL: Uniform Resource Locator – a URI that specifies the location of a resource and the protocol to retrieve it.
  • URN: Uniform Resource Name – a URI that identifies a resource by name in a particular namespace, without providing a location.

Common Mistakes & Misconceptions

  • Misconception: “URL and URI are interchangeable.” – While often used interchangeably in casual conversation, URL is a subset of URI. Every URL is a URI, but not every URI is a URL.
  • Misconception: “URN is obsolete.” – URNs are still used in some contexts (e.g., XML namespaces, ISBN).

Real-World Application

In REST APIs, the term “URI” is often used when referring to the identifier of a resource, even though it is actually a URL. For example, “the URI of a customer” might be `/customers/5`.

6. Design Rules of Thumb for Controllers

Definition

The lecture concludes with practical rules of thumb for designing MVC applications, focusing on the separation of concerns between Model, View, and Controller. Key rules: (1) It should be possible to change views without the model ever knowing. (2) It should be possible to change the underlying storage model without views ever knowing. (3) Controllers/actions should generally never talk to a database directly – that is the Model’s responsibility.

Explanation

Motivation: These rules enforce clean separation and prevent the common anti‑patterns (e.g., fat controllers, anemic models).

First principles: The Model encapsulates data and business logic, so it is the only component that should contain SQL queries or ORM calls. The View only displays data and captures input events (forwarding to Controller). The Controller orchestrates: it receives input, calls Model methods, and selects a View. If a Controller directly queries the database, it breaks encapsulation and makes the application harder to maintain.

Mechanism: To adhere to these rules, always put data access logic inside Model classes (or repositories). Controllers should only call Model methods (e.g., `Student.find(id)`, `Student.create(params)`). Views should not contain SQL or complex business logic; they may contain presentation logic (loops, conditionals for display).

Concrete example: A bad practice: writing `db.execute("SELECT * FROM students")` inside a Controller action. Good practice: `Student.all` (where `Student` is a Model class that internally uses SQLAlchemy).

Properties & characteristics: (1) Loose coupling – changes in one component have minimal impact. (2) Testability – each component can be tested in isolation. (3) Maintainability – the codebase is easier to understand and modify.

Relationship to other concepts: These rules are the practical application of Separation of Concerns (from L5.1).

Variations & special cases: Some frameworks (like Flask) do not enforce these rules; it is up to the developer to follow them. In Rails, the rules are enforced by convention (ActiveRecord models handle database queries).

Significance: Following these rules is the difference between a maintainable application and a “big ball of mud”.

Exam relevance: AQ5.4 does not directly test these rules, but they are important for understanding good MVC design.

Intuition / Mental Model

Think of the Model as the chef in a restaurant. The Controller is the waiter. The waiter never cooks (does not talk to the database). The chef never serves the customer directly (does not generate HTML). The waiter only takes orders and delivers food. If you change the menu (View), the chef doesn’t care. If you change the kitchen equipment (storage), the waiter doesn’t care.

Key Definitions

  • Fat Controller anti‑pattern: Putting too much business logic and data access code in Controllers.
  • Anemic Model anti‑pattern: Models that only contain data fields with no business logic, leaving all logic to Controllers.

Step-by-Step Procedure

  1. Step 1: Before writing a Controller action, ask: “Is this logic about data validation or business rules?” If yes, move it to the Model.
  2. Step 2: If you find yourself writing SQL in a Controller, refactor it into a Model method.
  3. Step 3: If you need to change the View, only modify template files; the Model should remain unchanged.
  4. Step 4: If you need to change the database (e.g., switch from SQLite to PostgreSQL), only modify the Model layer (e.g., update connection string, adjust queries). The Controllers and Views should stay the same.

Common Mistakes & Misconceptions

  • Misconception: “It’s okay to put simple SQL in Controllers for performance.” – This quickly leads to technical debt; use Model methods even for simple queries.
  • Misconception: “Views and Controllers are always tightly coupled, so it’s fine to put controller logic in views.” – Views should not contain business logic or database calls; they should only format data received from the Controller.

Real-World Application

Well‑designed MVC applications (e.g., Basecamp) strictly follow these rules. Code reviews often check for violations like SQL in Controllers.

Exam Tips & Points to Remember

  • Most frequently tested aspect: “Controllers should never talk to a database directly.”
  • Typical exam trap: Believing that the Controller can “temporarily” bypass the Model for performance – not acceptable in good design.

Key Takeaways

* Change the View without affecting the Model – this is possible because the Controller mediates.
* Change the storage model without affecting Views – the Model abstracts the database.
* Controllers must never directly execute SQL queries or talk to the database.
* Following these rules ensures loose coupling and maintainability.

Inter-Concept Connections

This lecture connects the abstract concept of “controller” from L5.1 with concrete implementation patterns. The resource controller (Laravel example) shows how actions are grouped in practice. The mapping of HTTP verbs to actions reinforces material from L5.2. The design rules tie back to Separation of Concerns.
